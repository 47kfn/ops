# AIエージェント運用ハーネス設計書（統合版）

## 0. この文書の位置づけ

この文書は、SWE_P1 / SWE_P2 / SWE_P3 を AI エージェント中心で並列推進するための「運用ハーネス」を、1枚のファイルで俯瞰できるようにまとめた統合設計書である。

**この文書の保存場所：** `ops/harness_design_master.md`（全プロジェクト共通の運用基盤として ops 配下に置く）

ここでいう「ハーネス」とは、単なる実装環境や CLI 設定だけではなく、次の要素をまとめた運用の土台を指す。

- どの AI が何を担当するか
- どのタスクをどの単位で渡すか
- どこに状態を残すか
- セッションが切れてもどう再開するか
- 途中経過・判断理由・失敗履歴をどこに残すか
- 破壊的操作や課金事故をどう防ぐか
- 人間がどのタイミングで介入するか

この文書は、現時点で確定している内容だけでなく、まだ未決定だが今後設計判断が必要になる項目も含めて記載する。
そのため、確定事項・暫定事項・未決定事項が混在する。重要なのは、未決定のまま放置することではなく、「何が決まっていて、何がまだ決まっていないか」を見える状態にすることである。

---

## 1. このハーネスの目的

### 1-1. 主目的

このハーネスの目的は、AI エージェントを安全に・長時間・再現性高く・観測可能な形で使いこなし、ユーザーが不在の時間帯（就寝中・仕事中）にも、開発タスクが前進し続ける状態を作ることである。

### 1-2. 具体的に達成したい状態

- ユーザーが毎回ゼロから説明しなくても、AI が現在地を把握できる
- 大きなタスクをそのまま渡すのではなく、途中で止まりにくい単位に分解できる
- 並列で複数の AI を動かしても、誰が何をしているか追跡できる
- セッションが切れても、ログと状態ファイルから再開できる
- 仕様変更や計画変更が起きても、どこを直せば良いかすぐ分かる
- ツール追加や運用変更があっても、構造を大きく壊さずに拡張できる
- 事故の可能性がある操作（削除、外部送信、課金が発生する操作など）を制御できる

### 1-3. このハーネスが対象とする範囲

対象に含めるもの：

- Claude Code / Codex / Antigravity / 必要に応じて Cursor / Gemini CLI などの役割分担
- Layer1・Layer2・Layer3 の管理構造
- セッション運用（tmux、再開性、ログ）
- ドキュメント管理
- Issue 管理
- 進捗管理
- 変更管理
- 安全設計
- コスト管理
- 失敗時の再開フロー

対象に含めないもの：

- 個別機能の詳細実装そのもの
- 最終的なプロダクト仕様の断定
- 本番構成の最終決定

## 2. 基本設計思想

### 2-1. AI が迷わない構造を優先する

人間にとって分かりやすいことも重要だが、このハーネスでは特に「AI が次に何をすべきか判断しやすいこと」を優先する。
そのため、ドキュメントや状態の保存方法は、見た目の美しさよりも以下を重視する。

- 正本が明確であること
- 状態が最新に更新されていること
- 依存関係が追えること
- 差分が小さく追跡しやすいこと
- 再開時に必要な情報だけを読めること

### 2-2. 1つの巨大コンテキストに頼らない

長時間運用では、1セッションに全てを詰め込む方式は不安定になりやすい。
そのため、次の考え方を採用する。

- 全体方針は上位ドキュメントに置く
- 個別タスクは Issue 単位に切る
- 実行ログは別保存する
- 再開時は必要ファイルだけ読む

### 2-3. 「会話」ではなく「状態ファイル」で継続する

AI の会話履歴だけに依存すると、セッション切断やコンテキスト圧縮で再現性が落ちる。
そのため、継続に必要な情報は原則としてリポジトリ内の状態ファイルに残す。

### 2-4. 正本は少なくする

同じことが複数の書類に書かれている状態は、変更時に必ず破綻する。
したがって、各種類の情報ごとに「正本」を定義し、それ以外は参照・要約・ミラーに留める。

## 3. 想定する全体構造

管理構造は大きく 3 層で構成する。

1. Layer1：全体管理AI
2. Layer2：実行管理AI
3. Layer3：実行AI

必要に応じて、人間（ユーザー）は全層の上位に位置する最終意思決定者として介入する。

---

## 4. 役割分担（誰が何をやるか）

## 4-1. ユーザー（最終意思決定者）

### 役割

- 最終的な方向性を決める
- 優先順位を確定する
- 重要な仕様変更を承認する
- 破壊的操作や課金が伴う操作の最終判断を行う
- 層間のつなぎ目で必要な修正・再指示を行う

### 基本姿勢

- 全部を自分で管理するのではなく、Layer1 に委譲する
- ただし完全放任ではなく、重大な方針・権限・予算・安全性には介入する
- なるべく「次に何を判断すればいいか」が分かる形で AI から報告を受ける

### 人間が主に見るもの

- project_overview
- roadmap
- status
- decisions
- 重要 issue の要約
- 重大ログ・失敗報告

---

## 4-2. Layer1：全体管理AI

### 役割

Layer1 は、全プロジェクトを俯瞰して「今どのタスクを進めるべきか」「どの層に何を渡すべきか」を判断する司令塔である。

### 主な責務

- プロジェクト全体の目的・範囲・優先順位を維持する
- フェーズやロードマップに照らして現在地を判断する
- 全体計画レベルの調査・計画を行う
- 新しい課題・ボトルネック・未解決事項を見つける
- Layer2 に渡す Issue を作成・更新・クローズする
- 仕様変更や計画変更が起きたときに、上位文書を更新対象として整理する
- 各 Issue の優先度、依存関係、担当層を決める
- 「今は何をやるべきではないか」も明示する
- **Layer2 からの判定案（ready_for_close / continue / needs_followup / blocked）を受け取り、最終判断を行う**
- **継続するか・issueをクローズするか・新規 issue を立てるかを決定する**

### Layer1 が直接やらないこと

- 長い実装作業そのもの
- 詳細なコード修正
- 詳細なデバッグ試行
- 1つの Issue 内の微細な作業管理

### Layer1 が主に読む文書

- `project_overview.md`
- `requirements.md`
- `roadmap.md`
- `status.md`
- `decisions.md`
- Issue _index（一覧）
- Issue ファイル（詳細）
- 主要ログの要約（必要時のみ `runs/` を参照）

### Layer1 が主に書く / 更新する文書

- `status.md`
- `decisions.md`
- Issue 新規作成 / 優先度変更 / 状態変更
- 必要に応じて `roadmap.md` の改訂案

---

## 4-3. Layer2：実行管理AI

### 役割

Layer2 は、Layer1 が作成した Issue を読んで、実行可能な単位に整形し、Layer3 実行 AI に渡し、途中経過・詰まり・完了条件の達成状況を管理する中間管理層である。

### 主な責務

- Issue の背景・目的・完了条件を読み取る
- issue 実行レベルの調査・計画を行う
- 実行可能なサブタスクに分割・整形する
- Layer3 に渡すプロンプトや作業指示を作成する
- 実行結果を確認し、Issue に反映する（Issue ファイルの現状を更新）
- 試行ログをもとに、次に試すべき手順を整理する
- ブロック要因があれば Layer1 またはユーザーにエスカレーションする
- **実行結果に対して判定案を作成し、Layer1 に提案する**
  - `ready_for_close`：完了条件を満たした
  - `continue_current_issue`：続行が必要
  - `needs_followup`：別 issue の作成が必要
  - `blocked`：ブロッカーがあり継続できない
- 1つの大タスクを「夜間に回せる粒度」まで落とし込む

### Layer2 が直接やらないこと

- 原則として大規模な実装そのものは担わない
- 全体方針の最終決定はしない
- 仕様変更の確定はしない
- **最終的な継続/終了/新規 issue の判断はしない（Layer1 に委ねる）**

### Layer2 が主に読む文書

- 担当 Issue ファイル
- 関連 `runs/` ログ
- 関連ドキュメント（要求仕様、設定メモ、運用ルール）
- `status.md` の自分に関係する部分

### Layer2 が主に書く / 更新する文書

- Issue の現在状態（issue ファイルを更新）
- 引き継ぎメモ
- 実行手順メモ
- サブタスク整理
- Layer3 用プロンプト案
- 判定案（Layer1 への提案）
- ブロッカー報告

---

## 4-4. Layer3：実行AI

### 役割

Layer3 は、実際に調査・実装・検証・修正・テスト・ログ出力を行う実働層である。

### 主な責務

- 指示されたスコープ内でコードや設定を変更する
- 実装・検証のための局所調査を行う
- コマンド実行・テスト実行・確認作業を行う
- 結果をログとして `runs/` に残す
- 失敗時に原因仮説と次の候補を残す
- 指示スコープを超える重大判断は行わない
- **結果を Layer2 に報告する**

### Layer3 が直接 Layer1 またはユーザーへ上げてよい例外ケース

通常は Layer2 → Layer1 の順でエスカレーションするが、以下の場合は Layer3 が直接上位へ報告すること：

- **課金リスク**：想定外の API 呼び出し・高コスト処理が発生しそうな場合
- **データ削除リスク**：取り返しのつかないデータ削除が発生しそうな場合
- **セキュリティ問題**：秘匿情報の露出・認証情報の漏洩リスクがある場合
- **致命的ブロッカー**：環境が壊れており全体進行が止まる場合

### Layer3 が主に読むもの

- Layer2 が作成したプロンプト / 手順書
- 対象 Issue
- 最低限必要な関連ファイル

### Layer3 が主に書くもの

- 実行ログ（`runs/` に出力）
- 作業結果サマリー
- エラーメモ
- 変更内容

---

## 5. 調査・計画の層別分担

調査と計画は各層で行うが、スコープが異なる。混在させないよう以下を原則とする。

| 層 | 調査・計画のスコープ |
|---|---|
| Layer1 | 全体計画レベル。プロジェクト全体の方向性・優先順位・フェーズ判断。 |
| Layer2 | issue 実行レベル。担当 issue の背景確認・実行手順の整理・サブタスク分解。 |
| Layer3 | 実装・検証レベル。コードの局所的調査・テスト実行・エラー原因の特定。 |

Layer3 が全体方針の調査を始めたら、スコープオーバー。Layer2 に戻す。

---

## 6. 実行単位の原則

### 6-1. 基本単位

- **1 issue = 1 Layer2 = 1 Layer3**
- 1つの issue に対して、1つの Layer2 が管理し、1つの Layer3 が実行する
- 現段階では複数の Layer3 を並列で走らせる運用はしない

### 6-2. issue が大きすぎる場合

- issue を分割する
- 分割後も「1 issue = 1 Layer2 = 1 Layer3」の原則を維持する
- 分割の判断は Layer1 が行う

---

## 7. タスクの流れ（基本ワークフロー）

```
[ユーザーまたは Layer1] 課題を認識
         ↓
[Layer1] Issue を定義・優先度・依存関係を決定
         ↓
[Layer2] Issue を読み、実行単位へ整形・Layer3 用プロンプト作成
         ↓
[Layer3] 実行・ログ出力（runs/ に保存）・結果報告
         ↓
[Layer2] 結果を要約・Issue を更新・判定案を作成
         ↓
[Layer1] 判定案を受け取り、最終判断
         ├─ ready_for_close   → Issue を close
         ├─ continue          → Layer2 に続行指示
         ├─ needs_followup    → 新規 Issue を作成
         └─ blocked           → ユーザーにエスカレーション
         ↓
[Layer1] status.md を更新
```

この流れの重要ポイントは、各層が前の層の代わりを完全にするのではなく、責務を分けることにある。

---

## 8. 判定フロー（Layer2 → Layer1）

Layer2 は実行結果の確認後、必ず以下のいずれかの判定案を Layer1 に提示する。

| 判定 | 意味 | Layer1 のアクション |
|---|---|---|
| `ready_for_close` | 完了条件をすべて満たした | Issue を closed/ へ移動 |
| `continue_current_issue` | 完了条件は未達・続行が有効 | Layer2 に続行指示 |
| `needs_followup` | 完了したが、次の作業が必要 | 新規 Issue を作成 |
| `blocked` | 続行できないブロッカーがある | ユーザーにエスカレーション |

Layer2 は判定案を提示するのみで、最終決定は Layer1 が行う。

## 11. Issue 管理の設計

### 11-1. Issue はファイルで管理する

AI 運用上、ローカルの Issue ファイルを正本にする。

理由：

- AI がローカルでまとめて読める
- 再開しやすい
- Git で追跡しやすい
- 1つのタスク単位で差分がきれいになる
- 並列実行時に衝突が少ない

### 11-2. 基本単位

- 1 issue = 1 ファイル
- 1 ファイル = 1 つの明確な完了条件
- 大きすぎるタスクは分割する

### 11-3. フォルダ構造

```text
00-docs/
  issue-management/
    _index.md          ← open 一覧・優先順位・blocked 一覧
    open/              ← 未完了・進行中・保留・ブロック中
      ISS-001-xxx.md
    closed/            ← 完了・中止・不要
      ISS-000-xxx.md
    runs/              ← 実行 AI の生ログ（Layer3 が出力）
      ISS-001/
        2026-03-29.md
```

**フォルダの使い分け：**

| フォルダ | 入れるもの |
|---|---|
| `open/` | 未完了・進行中・保留・ブロック中のすべて |
| `closed/` | 完了・中止・不要になったもの |
| `runs/` | Layer3 が出力する実行生ログ（詳細履歴） |

### 11-4. 命名規則

推奨形式：

- `ISS-001-login-screen-error.md`
- `ISS-002-n8n-local-setup.md`

ルール：

- 固定 ID を付ける
- slug は短く意味が伝わるものにする
- 日付はファイル名に入れない
- タイトル変更があってもファイル名はなるべく維持する

### 11-5. Issue ファイルとログの関係

| 種類 | 役割 | 主な読者 |
|---|---|---|
| issue ファイル | **現在の状態・要約・引き継ぎ情報の正本** | Layer1・Layer2 |
| `runs/` ログ | **実行詳細・試行履歴・生ログ** | Layer2（必要時）・Layer3 |

- **Layer1 は基本的に issue ファイルのみ読む**
- runs/ ログは必要時のみ参照する
- issue ファイルに試行の全詳細を書き込みすぎない

### 11-6. Issue ファイルの推奨要素

- ID
- タイトル
- 状態
- 優先度
- 依存関係
- 背景
- 目的
- 完了条件
- 現在状態
- やること
- 引き継ぎメモ
- 関連ログ（runs/ への参照）

### 11-7. `_index.md`

役割：

- open Issue 一覧
- 優先順位一覧
- blocked 一覧
- 直近注力ポイント

これにより、Layer1 が毎回全 Issue を開かなくても状況を俯瞰できる。

## 13. ドキュメントの更新ルール

### 13-1. 上位から下位へ更新する

変更が起きた場合は、原則として次の順で更新する。

1. `project_overview.md`（必要なら）
2. `requirements.md`
3. `roadmap.md`
4. `decisions.md`
5. `status.md`
6. 関連 Issue
7. 関連ログ / 実行手順

Issue だけ先に書き換える運用は、上位方針とのズレを生みやすいため避ける。

### 13-2. 日常更新と構造更新を分ける

**日常更新（頻繁に行う）：**

- `status.md`
- Issue の現在状態
- 実行ログ（runs/）

**構造更新（方針変更時のみ）：**

- `project_overview.md`
- `requirements.md`
- `roadmap.md`
- 重要判断としての `decisions.md`

### 13-3. 詳細を上位文書に詰め込みすぎない

- 上位文書は「判断に必要な情報」に絞る
- 試行錯誤の詳細は Issue と runs/ へ逃がす
- 再開時に読み込むファイル数を最小化する

---

## 14. 仕様変更・計画変更の扱い

仕様変更や計画変更は前提として発生するものと考える。
問題は変更の有無ではなく、反映の順序と記録方法である。

### 14-1. 変更時の基本手順

1. 何を変えるのかを明文化する
2. 変更理由を `decisions.md` に記録する
3. 上位文書を更新する
4. 影響を受ける Issue を洗い出す
5. 不要になった Issue は close または分割する
6. `status.md` を更新する
7. Layer2・Layer3 への指示を更新する

### 14-2. 変更理由として残すべきもの

- なぜ元の方針を変えるのか
- 何を優先するために変えるのか
- 影響を受ける機能・工程・ドキュメントは何か
- どの代替案を捨てたのか

---

## 15. セッション運用と再開性

このハーネスでは、AI セッションが一度で終わることを前提にしない。
むしろ、切断・中断・再接続・再開が発生する前提で設計する。

### 15-1. 基本方針

- セッション継続は tmux で担保する
- ただし tmux だけで再開性を担保したとは見なさない
- 再開可能性は必ずログと状態ファイルで担保する

### 15-2. 再開に必要な最低要素

- 対象 Issue
- 最新状態
- 直近の runs/ ログ
- 次にやるべき一手
- 失敗時の仮説

### 15-3. 望ましい状態

ユーザーや別 AI が途中参加しても、次の 3 点を見れば再開できること。

- `status.md`
- 該当 Issue ファイル
- 直近 runs/ ログ

---

## 16. ログ設計（runs/）

### 16-1. なぜログが重要か

長時間運用では、成功したこと以上に「何を試して失敗したか」を残さないと、毎回同じ失敗を繰り返す。

### 16-2. ログに残すべき最低項目

- 日時
- Issue ID
- 実行主体（どの AI / どのセッションか）
- 実行内容
- 成功 / 失敗 / 未確認
- 主要出力
- 次アクション候補

### 16-3. ログの粒度

- コマンド全出力をそのまま全部残す必要はない
- 重要ポイントを抽出したサマリーも残す
- 必要なら raw log への参照を別で持つ

---

## 17. 安全設計

AI エージェント運用では、実装速度だけでなく、事故防止をハーネスの一部として設計する必要がある。

### 17-1. 重点リスク

- ファイル削除・移動
- 秘密情報の露出
- 外部送信
- 勝手な push
- 課金が発生する操作
- 本番系リソースへの接続

### 17-2. 基本ルール

- 破壊的操作は原則確認を要する
- `.env` などの秘匿情報は読み取り対象から原則除外する
- 外部 API 呼び出しは用途と権限を明示する
- 課金が伴う可能性がある操作は、事前に実行条件を決める
- 本番系・公開系・課金系はローカル開発と明確に分ける

### 17-3. ドキュメントとして残すべき安全情報

- 何が禁止か
- 何は確認後なら許可か
- 何は自動で良いか
- 危険操作の例
- 環境ごとのルール差分

※ この安全ルールは、将来的に別紙 `safety_rules.md` として分離する可能性が高い。

## 18. コスト管理

複数ツールを並列で使う前提では、コストもハーネス設計の一部である。

### 18-1. 観点

- どのツールが subscription 範囲か
- API 課金が別に走る可能性があるか
- 夜間実行で無駄に長時間回り続けないか
- 重いモデルをどの層で使うか

### 18-2. 基本方針

- Layer1 は重い判断なので高品質モデルも許容
- Layer3 の反復試行は、必要以上に高コストモデルへ寄せない
- ツール追加時は「費用」「上限」「想定使用場面」を記録する

※ 将来的には `tool_policy.md` のような別紙管理が望ましい。

## 23. 現時点の運用結論（暫定）

1. 管理構造は「ユーザー + Layer1 + Layer2 + Layer3」の 4 者構造で整理する
2. Layer1 は司令塔、Layer2 は実行管理、Layer3 は実働に分ける
3. 調査・計画は層ごとにスコープを分ける（全体計画 / issue レベル / 局所実装）
4. **1 issue = 1 Layer2 = 1 Layer3 を原則とし、複数 Layer3 の並列は現段階では不使用**
5. 長時間運用の前提として、会話履歴ではなく状態ファイルを正本にする
6. ドキュメントは最小限に絞るが、正本は明確にする
7. 個別タスクは 1 issue = 1 file で管理する
8. Issue は `open/closed/runs` を分け、`_index.md` で全体俯瞰する
9. **issue ファイルは現状の要約、runs/ は実行の詳細、と明確に分担する**
10. 変更は上位文書から下位へ反映する
11. 再開性は tmux だけでなく `status + issue + runs` で担保する
12. **Layer2 は必ず判定案（ready_for_close / continue / needs_followup / blocked）を Layer1 に提示する**
13. **Layer3 は課金・削除・セキュリティ・致命的ブロッカーの場合のみ直接上位へ報告する**
14. ハーネス設計書は `ops/` 配下、プロジェクト文書は各プロジェクト内に置く
15. 安全・コスト・権限もハーネスの一部として明文化対象にする
