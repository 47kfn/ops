# AIエージェント運用ハーネス設計書（統合版）

## 0. この文書の位置づけ

この文書は、AI エージェント中心でプロジェクトを推進するための「運用ハーネス」を、1枚のファイルで俯瞰できるようにまとめた統合設計書である。

**この文書の保存場所：** `ops/harness_design_master.md`（全プロジェクト共通の運用基盤として ops 配下に置く）

**この文書の読者：** ユーザー（人間）。AI が直接この文書を読むことは想定しない。AI への指示は、本書の内容をもとに各 Layer 用の CLAUDE.md やスコープ設定として反映する。

ここでいう「ハーネス」とは、単なる実装環境や CLI 設定だけではなく、次の要素をまとめた運用の土台を指す。

- どの AI が何を担当するか
- どのタスクをどの単位で渡すか
- どこに状態を残すか
- セッションが切れてもどう再開するか
- 途中経過・判断理由・失敗履歴をどこに残すか
- 破壊的操作や課金事故をどう防ぐか
- 人間がどのタイミングで介入するか

確定事項・暫定事項・未決定事項が混在する。重要なのは、「何が決まっていて、何がまだ決まっていないか」を見える状態にすることである。

---

## 1. このハーネスの目的

### 1-1. 主目的

AI エージェントを安全に・長時間・再現性高く・観測可能な形で使いこなし、ユーザーが不在の時間帯（就寝中・仕事中）にも、開発タスクが前進し続ける状態を作ること。

### 1-2. 具体的に達成したい状態

- ユーザーが毎回ゼロから説明しなくても、AI が現在地を把握できる
- 大きなタスクを途中で止まりにくい単位に分解できる
- セッションが切れても、ログと状態ファイルから再開できる
- 仕様変更や計画変更が起きても、どこを直せば良いかすぐ分かる
- 事故の可能性がある操作（削除、外部送信、課金が発生する操作など）を制御できる

### 1-3. このハーネスが対象とする範囲

対象に含めるもの：

- Claude Code を中心としたレイヤー構造
- Layer1・Layer2・Layer3 の管理構造
- セッション運用（tmux、再開性、ログ）
- ドキュメント管理
- Issue 管理
- 進捗管理
- 変更管理
- 安全設計
- コスト管理
- 失敗時の再開フロー

対象に含めないもの：

- 個別機能の詳細実装そのもの
- 最終的なプロダクト仕様の断定
- 本番構成の最終決定

### 1-4. ツール方針

**現時点のメインツール：Claude Code（Claude Pro サブスク範囲内）**

ハーネス設計が安定し運用が軌道に乗った段階で、Codex やその他ツールへの適用拡張を検討する。本書では Claude Code 前提で記述する。

## 2. 基本設計思想

### 2-1. AI が迷わない構造を優先する

人間にとって分かりやすいことも重要だが、特に「AI が次に何をすべきか判断しやすいこと」を優先する。

- 正本が明確であること
- 状態が最新に更新されていること
- 依存関係が追えること
- 差分が小さく追跡しやすいこと
- 再開時に必要な情報だけを読めること

### 2-2. 1つの巨大コンテキストに頼らない

長時間運用では、1セッションに全てを詰め込む方式は不安定になりやすい。

- 全体方針は上位ドキュメントに置く
- 個別タスクは Issue 単位に切る
- 実行ログは別保存する
- 再開時は必要ファイルだけ読む
- コンテキスト肥大時は `/clear` で定期的にリセットし、状態ファイルから復帰する

### 2-3. 「会話」ではなく「状態ファイル」で継続する

AI の会話履歴だけに依存すると、セッション切断やコンテキスト圧縮で再現性が落ちる。
継続に必要な情報は原則としてリポジトリ内の状態ファイルに残す。

### 2-4. 正本は少なくする

同じことが複数の書類に書かれている状態は、変更時に必ず破綻する。
各種類の情報ごとに「正本」を定義し、それ以外は参照・要約・ミラーに留める。

## 3. 想定する全体構造

管理構造は大きく 3 層 + ユーザーで構成する。

1. **ユーザー**：最終意思決定者
2. **Layer1**：全体管理AI（当面はユーザー + AI の半自動運用）
3. **Layer2**：実行管理AI
4. **Layer3**：実行AI

---

## 4. 役割分担（誰が何をやるか）

### 4-1. ユーザー（最終意思決定者）

#### 役割

- 最終的な方向性を決める
- 優先順位を確定する
- 重要な仕様変更を承認する
- 破壊的操作や課金が伴う操作の最終判断を行う
- 外部 API 利用の承認を行う
- 層間のつなぎ目で必要な修正・再指示を行う

#### 基本姿勢

- 全部を自分で管理するのではなく、Layer1 に委譲する
- ただし完全放任ではなく、重大な方針・権限・予算・安全性には介入する
- なるべく「次に何を判断すればいいか」が分かる形で AI から報告を受ける

#### 人間が主に見るもの

- project_overview
- roadmap
- status
- decisions
- 重要 issue の要約
- 重大ログ・失敗報告

---

### 4-2. Layer1：全体管理AI

#### 実装方針

- **当面はユーザー + Claude Code の半自動運用**とする
- ユーザーの技術力・経験の向上に応じて、委譲の割合を段階的に高める
- 常駐セッションとするが、コンテキスト肥大化を防ぐため定期的に `/clear` を実行する
- `/clear` 後は `status.md` → `_index.md` → 直近の重要 issue の順に読み込んで状態を復元する
- **単一プロジェクトを対象とする**（複数プロジェクト横断は現段階では前提としない）

#### 役割

Layer1 は、プロジェクトを俯瞰して「今どのタスクを進めるべきか」「どの層に何を渡すべきか」を判断する司令塔である。

#### 主な責務

- プロジェクト全体の目的・範囲・優先順位を維持する
- フェーズやロードマップに照らして現在地を判断する
- 全体計画レベルの調査・計画を行う
- 新しい課題・ボトルネック・未解決事項を見つける
- Layer2 に渡す Issue を作成・更新・クローズする
- 仕様変更や計画変更が起きたときに、上位文書を更新対象として整理する
- 各 Issue の優先度、依存関係、担当層を決める
- 「今は何をやるべきではないか」も明示する
- Layer2 からの判定案（ready_for_close / continue / needs_followup / blocked）を受け取り、最終判断を行う
- 継続するか・issue をクローズするか・新規 issue を立てるかを決定する
- `_index.md` を更新する（最終責任はユーザー）
- **Layer2 が判定案を出していない issue がないか定期的に確認する**

#### Layer1 が直接やらないこと

- 長い実装作業そのもの
- 詳細なコード修正
- 詳細なデバッグ試行
- 1つの Issue 内の微細な作業管理

#### Layer1 が主に読む文書

- `project_overview.md`
- `requirements.md`
- `roadmap.md`
- `status.md`
- `decisions.md`
- Issue `_index.md`（一覧）
- Issue ファイル（詳細）
- 主要ログの要約（必要時のみ `runs/` を参照）

#### Layer1 が主に書く / 更新する文書

- `status.md`
- `decisions.md`
- `_index.md`
- Issue 新規作成 / 優先度変更 / 状態変更
- 必要に応じて `roadmap.md` の改訂案

---

### 4-3. Layer2：実行管理AI

#### 役割

Layer2 は、Layer1 が作成した Issue を読んで、実行可能な単位に整形し、Layer3 実行 AI に渡し、途中経過・詰まり・完了条件の達成状況を管理する中間管理層である。

#### Layer間のインターフェース

- **Layer1 → Layer2**：Layer2 の Claude Code セッションを起動し、対象の issue ファイルパスを渡してプロンプトを入力する
- **Layer2 → Layer1**：issue ファイル内の `## 判定案` セクションに追記する形で報告する
- **Layer2 → Layer3**：Layer3 の Claude Code セッションを起動し、作業指示プロンプトを入力する
- **Layer3 → Layer2**：issue ファイル内の `## 実行結果` セクションに追記する形で報告する

#### 主な責務

- Issue の背景・目的・完了条件を読み取る
- issue 実行レベルの調査・計画を行う
- 実行可能なサブタスクに分割・整形する
- Layer3 に渡すプロンプトや作業指示を作成する
- 実行結果を確認し、Issue に反映する（Issue ファイルの現状を更新）
- 試行ログをもとに、次に試すべき手順を整理する
- ブロック要因があれば Layer1 またはユーザーにエスカレーションする
- **外部 API の利用が必要な場合は、必ずユーザーの承諾を得る。承諾後もレート制限・コスト上限を設定する**
- 実行結果に対して判定案を作成し、Layer1 に提案する
  - `ready_for_close`：完了条件を満たした
  - `continue_current_issue`：続行が必要
  - `needs_followup`：別 issue の作成が必要
  - `blocked`：ブロッカーがあり継続できない
- 1つの大タスクを「夜間に回せる粒度」まで落とし込む

#### Layer2 が直接やらないこと

- 原則として大規模な実装そのものは担わない
- 全体方針の最終決定はしない
- 仕様変更の確定はしない
- 最終的な継続/終了/新規 issue の判断はしない（Layer1 に委ねる）

#### Layer2 が主に読む文書

- 担当 Issue ファイル
- 関連 `runs/` ログ
- 関連ドキュメント（要求仕様、設定メモ、運用ルール）
- `status.md` の自分に関係する部分

#### Layer2 が主に書く / 更新する文書

- Issue の現在状態（issue ファイルの `## 現在状態` を更新）
- Issue の判定案（issue ファイルの `## 判定案` を更新）
- 引き継ぎメモ
- 実行手順メモ
- サブタスク整理
- Layer3 用プロンプト案
- ブロッカー報告

---

### 4-4. Layer3：実行AI

#### 役割

Layer3 は、実際に調査・実装・検証・修正・テスト・ログ出力を行う実働層である。

#### 主な責務

- 指示されたスコープ内でコードや設定を変更する
- 実装・検証のための局所調査を行う
- コマンド実行・テスト実行・確認作業を行う
- 結果をログとして `runs/` に残す
- 失敗時に原因仮説と次の候補を残す
- 指示スコープを超える重大判断は行わない
- 結果を Layer2 に報告する（issue ファイルの `## 実行結果` セクションに追記）

#### Layer3 が直接 Layer1 またはユーザーへ上げてよい例外ケース

通常は Layer2 → Layer1 の順でエスカレーションするが、以下の場合は Layer3 が直接上位へ報告すること：

- **課金リスク**：想定外の API 呼び出し・高コスト処理が発生しそうな場合
- **データ削除リスク**：取り返しのつかないデータ削除が発生しそうな場合
- **セキュリティ問題**：秘匿情報の露出・認証情報の漏洩リスクがある場合
- **致命的ブロッカー**：環境が壊れており全体進行が止まる場合

**報告方法：** issue ファイルの先頭に `## ⚠ ESCALATION` セクションを追記し、理由と状況を記載する。Layer1 または ユーザーがこれを発見して対応する。

#### Layer3 が主に読むもの

- Layer2 が作成したプロンプト / 手順書
- 対象 Issue
- 最低限必要な関連ファイル

#### Layer3 が主に書くもの

- 実行ログ（`runs/` に出力）
- 作業結果サマリー（issue ファイルの `## 実行結果` セクション）
- エラーメモ
- 変更内容

## 5. 調査・計画の層別分担

調査と計画は各層で行うが、スコープが異なる。混在させないよう以下を原則とする。

| 層 | 調査・計画のスコープ |
|---|---|
| Layer1 | 全体計画レベル。プロジェクト全体の方向性・優先順位・フェーズ判断。 |
| Layer2 | issue 実行レベル。担当 issue の背景確認・実行手順の整理・サブタスク分解。 |
| Layer3 | 実装・検証レベル。コードの局所的調査・テスト実行・エラー原因の特定。 |

Layer3 が全体方針の調査を始めたら、スコープオーバー。Layer2 に戻す。

---

## 6. 実行単位の原則

### 6-1. 基本単位

- **1 issue = 1 Layer2 セッション = 1 Layer3 セッション**
- 1つの issue に対して、1つの Layer2 が管理し、1つの Layer3 が実行する
- 複数の issue を並列で進めることは可能（それぞれ独立したセッション）

### 6-2. issue が大きすぎる場合

- issue を分割する
- 分割後も「1 issue = 1 Layer2 = 1 Layer3」の原則を維持する
- 分割の判断は Layer1 が行う

### 6-3. 並列実行について

- 異なる issue 間の並列実行は許可する
- 1つの issue 内での複数セッション並列は現段階では行わない
- 将来的に Claude の agent team 等の機能で自然に並列化される可能性があり、その場合はハーネス側での手動並列管理は不要になる見込み

## 7. タスクの流れ（基本ワークフロー）

```
[ユーザーまたは Layer1] 課題を認識
         ↓
[Layer1] Issue を定義・優先度・依存関係を決定
         ↓
[ユーザーまたは Layer1] Layer2 の Claude Code セッションを起動、issue ファイルを指定
         ↓
[Layer2] Issue を読み、実行単位へ整形・Layer3 用プロンプト作成
         ↓
[Layer2] Layer3 の Claude Code セッションを起動、作業指示を入力
         ↓
[Layer3] 実行・ログ出力（runs/ に保存）・結果を issue ファイルに追記
         ↓
[Layer2] 結果を確認・Issue を更新・判定案を issue ファイルに追記
         ↓
[Layer1] 判定案を確認し、最終判断
         ├─ ready_for_close   → Issue を closed/ へ移動
         ├─ continue          → Layer2 に続行指示
         ├─ needs_followup    → 新規 Issue を作成
         └─ blocked           → ユーザーにエスカレーション
         ↓
[Layer1] status.md・_index.md を更新
```

---

## 8. 判定フロー（Layer2 → Layer1）

Layer2 は実行結果の確認後、必ず issue ファイル内の `## 判定案` セクションに以下のいずれかを記載する。

| 判定 | 意味 | Layer1 のアクション |
|---|---|---|
| `ready_for_close` | 完了条件をすべて満たした | Issue を closed/ へ移動 |
| `continue_current_issue` | 完了条件は未達・続行が有効 | Layer2 に続行指示 |
| `needs_followup` | 完了したが、次の作業が必要 | 新規 Issue を作成 |
| `blocked` | 続行できないブロッカーがある | ユーザーにエスカレーション |

Layer2 は判定案を提示するのみで、最終決定は Layer1 が行う。

## 11. Issue 管理の設計

### 11-1. Issue はファイルで管理する

AI 運用上、ローカルの Issue ファイルを正本にする。

理由：

- AI がローカルでまとめて読める
- 再開しやすい
- Git で追跡しやすい
- 1つのタスク単位で差分がきれいになる
- 並列実行時に衝突が少ない

### 11-2. 基本単位

- 1 issue = 1 ファイル
- 1 ファイル = 1 つの明確な完了条件
- 大きすぎるタスクは分割する

### 11-3. フォルダ構造

```text
00-docs/
  issue-management/
    _index.md          ← open 一覧・優先順位・blocked 一覧
    open/              ← 未完了・進行中・保留・ブロック中
      ISS-001-xxx.md
    closed/            ← 完了・中止・不要
      ISS-000-xxx.md
    runs/              ← 実行 AI の生ログ（Layer3 が出力）
      ISS-001/
        run-001_2026-03-29.md
        run-002_2026-03-30.md
```

**フォルダの使い分け：**

| フォルダ | 入れるもの |
|---|---|
| `open/` | 未完了・進行中・保留・ブロック中のすべて |
| `closed/` | 完了・中止・不要になったもの |
| `runs/` | Layer3 が出力する実行生ログ（詳細履歴） |

### 11-4. 命名規則

推奨形式：

- `ISS-001-login-screen-error.md`
- `ISS-002-n8n-local-setup.md`

ルール：

- 固定 ID を付ける
- slug は短く意味が伝わるものにする
- 日付はファイル名に入れない
- タイトル変更があってもファイル名はなるべく維持する

### 11-5. Issue ファイルとログの関係

| 種類 | 役割 | 主な読者 |
|---|---|---|
| issue ファイル | **現在の状態・要約・引き継ぎ情報の正本** | Layer1・Layer2 |
| `runs/` ログ | **実行詳細・試行履歴・生ログ** | Layer2（必要時）・Layer3 |

- **Layer1 は基本的に issue ファイルのみ読む**
- runs/ ログは必要時のみ参照する
- issue ファイルに試行の全詳細を書き込みすぎない

### 11-6. runs/ ログの命名と関連付け

同一 issue で複数回の試行がある場合：

- `runs/ISS-001/run-001_2026-03-29.md`（1回目）
- `runs/ISS-001/run-002_2026-03-30.md`（2回目）

ルール：

- ディレクトリ名は issue ID と一致させる
- ファイル名は `run-{連番}_{日付}.md` の形式
- issue ファイル内の `## 関連ログ` セクションにログファイルへの参照を記載する

### 11-7. Issue ファイルのテンプレート

```markdown
# ISS-{番号} {タイトル}

## メタ情報
- **状態**: open / in_progress / blocked / closed
- **優先度**: high / medium / low
- **依存**: なし / ISS-xxx
- **作成日**: YYYY-MM-DD
- **最終更新**: YYYY-MM-DD

## 背景
（なぜこの issue が必要なのか）

## 目的
（この issue で達成すること）

## 完了条件
- [ ] 条件1
- [ ] 条件2

## 現在状態
（Layer2 が随時更新。今どこまで進んでいるか）

## やること
（次のアクション）

## 引き継ぎメモ
（セッション切れ時に次の AI が読む情報）

## 実行結果
（Layer3 が追記。直近の実行結果サマリー）

## 判定案
（Layer2 が追記。ready_for_close / continue_current_issue / needs_followup / blocked + 理由）

## 関連ログ
- runs/ISS-{番号}/run-001_YYYY-MM-DD.md
```

### 11-8. `_index.md`

役割：

- open Issue 一覧
- 優先順位一覧
- blocked 一覧
- 直近注力ポイント

更新責任：Layer1 が更新する。最終責任者はユーザー。

これにより、Layer1 が毎回全 Issue を開かなくても状況を俯瞰できる。

## 13. ドキュメントの更新ルール

### 13-1. 上位から下位へ更新する

変更が起きた場合は、原則として次の順で更新する。

1. `project_overview.md`（必要なら）
2. `requirements.md`
3. `roadmap.md`
4. `decisions.md`
5. `status.md`
6. 関連 Issue
7. 関連ログ / 実行手順

Issue だけ先に書き換える運用は、上位方針とのズレを生みやすいため避ける。

### 13-2. 日常更新と構造更新を分ける

**日常更新（頻繁に行う）：**

- `status.md`
- Issue の現在状態
- 実行ログ（runs/）
- `_index.md`

**構造更新（方針変更時のみ）：**

- `project_overview.md`
- `requirements.md`
- `roadmap.md`
- 重要判断としての `decisions.md`

### 13-3. 状態ファイルの更新タイミング

`status.md` は以下のタイミングで必ず更新する：

- issue の新規作成時
- issue の状態変更時（in_progress, blocked, closed 等）
- issue の完了時
- ブロッカー発生時
- 日次の作業開始・終了時

乖離が発生した場合は手動で修正する。

### 13-4. 詳細を上位文書に詰め込みすぎない

- 上位文書は「判断に必要な情報」に絞る
- 試行錯誤の詳細は Issue と runs/ へ逃がす
- 再開時に読み込むファイル数を最小化する

## 14. decisions.md の運用基準

### 14-1. 記録する判断の基準

以下のいずれかに該当する判断を記録する：

- **技術選定**：ライブラリ、フレームワーク、アーキテクチャの選択
- **方針変更**：当初の計画から方向転換した場合
- **トレードオフ**：複数の選択肢から1つを選んだ場合（捨てた案も記録）
- **外部制約**：法規制、API 制限、コスト制約等により選択が強制された場合
- **スコープ変更**：機能の追加・削除・延期

### 14-2. 記録フォーマット

```markdown
### DEC-{番号}: {判断タイトル}
- **日付**: YYYY-MM-DD
- **判断**: {何を決めたか}
- **理由**: {なぜその判断をしたか}
- **却下案**: {検討したが採用しなかった案}
- **影響**: {この判断が影響する issue・文書}
```

### 14-3. 記録しないもの

- 実装の細かい技術的判断（変数名、関数分割等）→ コードとコミットメッセージで十分
- 自明な判断 → 記録のコストに見合わない

## 15. 仕様変更・計画変更の扱い

仕様変更や計画変更は前提として発生するものと考える。

### 15-1. 変更時の基本手順

1. 何を変えるのかを明文化する
2. 変更理由を `decisions.md` に記録する
3. 上位文書を更新する
4. 影響を受ける Issue を洗い出す
5. 不要になった Issue は close または分割する
6. `status.md` を更新する
7. Layer2・Layer3 への指示を更新する

### 15-2. 変更理由として残すべきもの

- なぜ元の方針を変えるのか
- 何を優先するために変えるのか
- 影響を受ける機能・工程・ドキュメントは何か
- どの代替案を捨てたのか

## 16. セッション運用と再開性

AI セッションが一度で終わることを前提にしない。切断・中断・再接続・再開が発生する前提で設計する。

### 16-1. 基本方針

- セッション継続は tmux で担保する
- ただし tmux だけで再開性を担保したとは見なさない
- 再開可能性は必ずログと状態ファイルで担保する
- コンテキスト肥大化時は `/clear` で定期的にリセットする

### 16-2. 再開に必要な最低要素

- 対象 Issue
- 最新状態
- 直近の runs/ ログ
- 次にやるべき一手
- 失敗時の仮説

### 16-3. 望ましい状態

ユーザーや別 AI が途中参加しても、次の 3 点を見れば再開できること。

- `status.md`
- 該当 Issue ファイル
- 直近 runs/ ログ

## 17. ログ設計（runs/）

### 17-1. なぜログが重要か

長時間運用では、成功したこと以上に「何を試して失敗したか」を残さないと、毎回同じ失敗を繰り返す。

### 17-2. ログのヘッダ（構造化フォーマット）

各ログファイルは以下のヘッダで始めること：

```markdown
# Run Log: ISS-{番号} / Run {連番}

- **日時**: YYYY-MM-DD HH:MM
- **Issue ID**: ISS-{番号}
- **実行主体**: Layer3 / Claude Code
- **セッション**: tmux セッション名（あれば）
- **結果**: success / failure / partial / aborted

## サマリー
（3行以内で結果を要約）

## 詳細
（実行内容・出力・エラー等）

## 次アクション候補
（失敗時は原因仮説と次に試すべきことを記載）
```

### 17-3. ログの粒度

- コマンド全出力をそのまま全部残す必要はない
- 重要ポイントを抽出したサマリーも残す
- 必要なら raw log への参照を別で持つ

---

## 18. 安全設計

AI エージェント運用では、実装速度だけでなく、事故防止をハーネスの一部として設計する。

### 18-1. 重点リスク

- ファイル削除・移動
- 秘密情報の露出
- 外部送信
- 勝手な push
- 課金が発生する操作
- 本番系リソースへの接続

### 18-2. 基本ルール

- 破壊的操作は原則確認を要する
- `.env` などの秘匿情報は読み取り対象から原則除外する
- 外部 API 呼び出しは用途と権限を明示し、**Layer2 で必ずユーザーの承諾を得る**
- 課金が伴う可能性がある操作は、事前に実行条件と上限を決める
- 本番系・公開系・課金系はローカル開発と明確に分ける
- **AI ツールは基本的にサブスクリプション範囲内で使用する**

### 18-3. Claude Code の権限設定

- Layer3 は原則として Claude Code の自動承認（auto-yes）を **使わない**
- 必要に応じて `--allowedTools` でスコープを絞る
- `CLAUDE.md` に禁止操作を明記する

### 18-4. ドキュメントとして残すべき安全情報

- 何が禁止か
- 何は確認後なら許可か
- 何は自動で良いか
- 危険操作の例
- 環境ごとのルール差分

※ 運用が成熟したら別紙 `safety_rules.md` として分離する。

## 19. コスト管理

### 19-1. 基本方針

- **全ツールをサブスクリプション範囲内で使用する**ことを原則とする
- 現在の契約：Claude Pro / ChatGPT Plus / Google AI Pro
- 開発中に外部 API を利用する場合は、必ず Layer2 でユーザー承諾を得た上で、レート制限・コスト上限を設定する

### 19-2. 観点

- サブスク範囲を超える API 課金が発生しないか
- 夜間実行で無駄に長時間回り続けないか
- 不要なリトライでトークンを消費していないか

### 19-3. Layer 別の考え方

- Layer1 は判断精度が重要なので、高品質モデルを使用（Claude Opus 等）
- Layer3 の反復試行は、タスクに応じてモデルを選択（Sonnet で十分なケースも多い）

## 24. 現時点の運用結論（暫定）

1. 管理構造は「ユーザー + Layer1 + Layer2 + Layer3」の 4 者構造
2. **Layer1 は当面ユーザー + Claude Code の半自動運用**。段階的に委譲を拡大
3. **ツールは Claude Code に一本化**。安定後に他ツールへ拡張
4. Layer間の情報伝達は **issue ファイルへの追記** を基本とする
5. Layer 起動は **Claude Code セッションの起動 + プロンプト入力** で行う
6. 調査・計画は層ごとにスコープを分ける（全体計画 / issue レベル / 局所実装）
7. **1 issue = 1 Layer2 = 1 Layer3** を原則とする
8. 異なる issue 間の並列実行は許可する
9. 長時間運用の前提として、会話履歴ではなく状態ファイルを正本にする
10. ドキュメントは最小限に絞るが、正本は明確にする
11. Issue は `open/closed/runs` を分け、`_index.md` で全体俯瞰する
12. issue ファイルは現状の要約、runs/ は実行の詳細、と明確に分担する
13. runs/ ログは `run-{連番}_{日付}.md` で管理し、issue から参照する
14. 変更は上位文書から下位へ反映する
15. 再開性は tmux だけでなく `status + issue + runs` で担保する
16. Layer2 は必ず判定案を issue ファイルに記載する
17. Layer3 は課金・削除・セキュリティ・致命的ブロッカーの場合のみ直接上位へ報告する
18. ハーネス設計書は `ops/` 配下（人間用）、AI 向けルールは各プロジェクトの `CLAUDE.md` に配置
19. 安全・コスト・権限もハーネスの一部として明文化対象にする
20. **外部 API 利用は Layer2 でユーザー承諾必須 + 制限設定必須**
21. Layer1 は定期的にヘルスチェック（整合性確認）を行う
