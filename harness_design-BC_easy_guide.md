# B案・C案ハーネス設計 超わかりやすい解説（非エンジニア向け）

## 0. 先に結論（30秒版）

- **B案**は「Claude運用を、GitHub中心で回す実務型」。
  - 仕事の管理は GitHub Projects / Issues。
  - 実作業は task pack と worktree で安全に分離。
  - 3つの役（Triage / Executor / Reviewer）で回す。
- **C案**は「Claude運用を、repo内の `.ops/queue` 中心で回す工場型」。
  - `proposed → ready → running → review → done` のような**明示キュー**で流す。
  - AI主導で次タスク候補を作り、人間は本数と優先度を決める。

ざっくり言うと、
- **B案 = GitHubを表の看板にした運用**
- **C案 = repo内キューを中枢にした運用**
です。

---

## 1. まず、A案との違いを一言で

あなたのA案は「統合設計書として全体像を俯瞰する“憲法”」です。

- A案: 何を目指すか、層の考え方、原則（思想）
- B案: Claude Code の機能（settings / skills / subagents / hooks）を前提にした、**実装しやすい構成**
- C案: `.ops/queue` を中心にした、**日々回しやすい運用導線**

つまり B/C は、A案を「現場で毎日回る形」に落とした実務版です。

---

## 2. B案を超わかりやすく

## 2-1. B案の本質

B案のキモはこれです。

> **役割を“会話で宣言”しない。環境で決める。**

「今から私はレビュー担当です」と毎回言わせるのではなく、
- どの task pack を読んでいるか
- どの worktree にいるか
- どの permissions（編集可/不可）で動いているか

で、AIの役割を固定します。

### 例えると
- 会話でお願いする運用 = 毎朝、同じ注意事項を口頭で言う職場
- B案の運用 = 入室カードと制服で役割が自動的に決まる職場

人間の言い忘れに依存しにくいのが強みです。

## 2-2. B案の構造（4層）

1. **GitHub Projects / Issues**（見える化・優先順位）
2. **repo内 task pack**（実行の正本）
3. **git worktree**（作業場の分離）
4. **Claude settings / permissions / hooks / skills**（行動制御）

重要ポイント：
- 「何をやるか」は task pack が正本
- 「どこでやるか」は worktree
- 「どう安全にやるか」は settings / permissions / hooks

## 2-3. B案の役割分担（3ロール）

- **Triage**: 次に何をやるか選ぶ
- **Executor**: 実装する
- **Reviewer**: 出来を検査する

これは別AIというより、**同じClaudeを別セッションで使い分ける**イメージです。

## 2-4. B案が向いているチーム

- GitHub運用がすでに定着している
- Issue/PRベースで進捗を追いたい
- 監査性（誰が何を決めたか）を重視したい

---

## 3. C案を超わかりやすく

## 3-1. C案の本質

C案のキモはこれです。

> **タスクは“キューの状態遷移”で流す。**

`.ops/queue/` にタスクを置き、
`inbox / proposed / ready / running / review / blocked / done`
の棚を移動させながら進めます。

### 例えると
- C案は工場のコンベア方式です。
- タスクという箱が、工程ごとに棚を移動する。
- どこで詰まっているか（blocked）も一目で分かる。

## 3-2. C案の運用思想

- AIが「次にやる候補」を作る（proposed）
- 人間は「今日はこれを回す」を決める（readyへ昇格）
- 実行AIが作業する（running）
- レビューAIが判定材料を作る（review）
- 人間が最終判断（done / 差し戻し）

つまり **AI主導 + 人間の最終統制** です。

## 3-3. C案が向いているチーム

- 日次で複数タスクを流す“運用業務”が多い
- 「滞留している場所」を可視化したい
- GitHubより先に、repo内で機械的に回したい

---

## 4. B案とC案の違い（迷わない比較）

- **中心となる盤面**
  - B案: GitHub Projects / Issues
  - C案: `.ops/queue` の状態棚
- **強み**
  - B案: 対外共有・PR連携・監査のしやすさ
  - C案: 日々の処理量に強い・詰まり可視化が速い
- **弱み**
  - B案: キュー運用が弱いと待ち行列が見えにくい
  - C案: GitHub同期を怠ると外部から見えにくい

実務では「C案で内部運用し、要所でB案（Issue/PR）に同期」が最も安定しやすいです。

---

## 5. 実際の具体的ワークフロー（非エンジニア版）

以下は、あなたが毎日回す時の手順です。

## 5-1. 朝（10〜20分）: 今日回す仕事を決める

1. `proposed` を見る（AIが作った候補）
2. 重要度と依存関係を見て、今日やる分だけ `ready` へ
3. 危険タスク（破壊的変更・課金・本番操作）が混じっていないか確認

ここは人間の意思決定ゾーンです。

## 5-2. 日中（実行）: 1タスクずつ着手

1. `ready` から1件選ぶ
2. そのタスク専用 worktree を作る（1 task = 1 worktree）
3. Executor Claude が実装
4. `RUNLOG` / `VERIFY` / `HANDOFF` を更新
5. `review` に回す

ポイント：
- 同時並行でも worktree が分かれているので衝突しにくい
- 失敗しても task単位で被害を閉じ込められる

## 5-3. 夕方（レビュー）: 出荷判定

1. Reviewer Claude が差分・検証結果を確認
2. 判定を返す
   - OK: done
   - 修正必要: ready または running に戻す
   - 要相談: blocked
3. 必要なら GitHub Issue / Project を更新

## 5-4. 夜（無人運転に寄せる）

1. 低リスクタスクだけ ready に残す
2. 危険操作は permissions でブロック済みにする
3. 翌朝、`running/review/blocked` をまとめて確認

---

## 6. すぐ使える「判断ルール」

- **人間が必ず判断するもの**
  - 優先順位
  - 破壊的変更
  - 外部送信/課金
  - 完了クローズ
- **AIに委譲しやすいもの**
  - 下調べ
  - タスク分解
  - 実装とテスト
  - 手順書更新

この線引きが明確だと、心理的安全性が一気に上がります。

---

## 7. あなた向けの実践アドバイス（A案理解者向け）

A案を既に把握しているあなたは、次の順で入れると理解が早いです。

1. **先にC案のキューだけ導入**
   - `proposed/ready/running/review/done` だけでも運用開始
2. **次にB案の settings/permissions を強化**
   - 危険操作を技術的に止める
3. **最後に GitHub 同期を整備**
   - 外向け可視化と監査を強化

この順番なら、最初から完璧を狙わずに“回る”状態へ到達しやすいです。

---

## 8. 1文でまとめると

- **B案**は「Claude運用を制度化して事故を減らす設計」。
- **C案**は「タスクを流れ作業にして停滞を減らす設計」。
- 2つは競合ではなく、**組み合わせると強い補完関係**です。
