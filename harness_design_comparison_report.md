# harness_design A/B/C 比較徹底分析（2026-04-02）

## 1. 評価の前提

本比較は、次の2観点を重視して評価した。

1. **将来性**
   - モデル性能の向上、Claude/Codex機能追加、運用環境変化が起きても設計の骨格が崩れないか
2. **非同期運用での意思決定力**
   - 就寝中・仕事中に実装を走らせ、起床時/帰宅時に「続行・やり直し・新規着手」を高精度にトリアージできるか

---

## 2. 3案の要点サマリ

### A案（Layer型ガバナンス重視）

- Layer1/2/3 を明確に分離し、責務境界・判定責任を制度化。
- source of truth を `status.md` / issueファイル / `runs/` に分散しつつ役割分担。
- 判定案（`ready_for_close` / `continue_current_issue` / `needs_followup` / `blocked`）を明示。
- Claude中心、Codex拡張は将来検討。

**強み:** 組織運用・責任分解・監査性。
**弱み:** 実行オペレーション（queue状態遷移、worktree強制、hook自動化）がB/Cより抽象。

### B案（Claude運用の実装可能性重視）

- Managed/User/Project/Local/Task scopeを具体化。
- `CLAUDE.md` は憲法、強制事項は settings/permissions/hooks へ分離。
- skills / subagents / hook lifecycle / command運用を詳細化。
- `1 task = 1 worktree` と handoff重視。

**強み:** Claude実装に直結、再現可能な運用手順が多い。
**弱み:** Claude依存度が高く、ベンダー抽象化はA/Cより弱い。

### C案（queue駆動・運用自動化重視）

- `proposed/ready/running/review/blocked/done` の状態機械を中心に設計。
- `.ops/queue` と `.ops/runs` を実行のtruthとして定義。
- triage/execution/reviewを役割セッション化し、hook/subagent/skillを運用フローに密着。
- 「将来のCodex等への展開でも骨格流用可」を明示。

**強み:** 非同期運用・朝夕トリアージの即応性が最も高い。
**弱み:** 運用導入コストが最も高く、初期整備（テンプレ・hook・queue discipline）が必須。

---

## 3. 評価軸1: 将来性（進化耐性）

## 3-1. A案

- **高評価ポイント**
  - ツールではなく責務（Layer）で設計しているため、モデル更新に強い。
  - 判定責任がLayer1に集中し、モデル差が出ても意思決定の最終責任がぶれにくい。
- **課題**
  - 実運用の強制装置（queue state machine、hooks、worktree隔離）が薄く、運用品質が人依存になりやすい。

**将来性評価:** 8.0/10（概念は強いが運用固定化がやや弱い）

## 3-2. B案

- **高評価ポイント**
  - Claudeの仕様（permissions/sandbox/hooks/skills/subagents）に沿っており、現実運用で崩れにくい。
  - 「指示文よりclient enforcementが強い」という原則が明確で、モデル挙動の揺れに強い。
- **課題**
  - Claude機能前提が強く、Codex等へ横展開する際は翻訳コストが発生。

**将来性評価:** 8.4/10（短中期で強い、マルチベンダー耐性は中程度）

## 3-3. C案

- **高評価ポイント**
  - queue/task packet/worktree/handoff を中心にしたため、モデルやエージェントが変わっても運用骨格を維持しやすい。
  - 役割を「会話」ではなく「状態・境界・権限」で定義しており、機能進化によるブレを吸収しやすい。
- **課題**
  - 成熟運用に達するまでの初期設計負荷が高い。

**将来性評価:** 9.2/10（最も進化耐性が高い）

---

## 4. 評価軸2: 非同期運用での判断/トリアージ精度

## 4-1. A案

- **できること**
  - `status + issue + runs` により再開性は確保。
  - Layer2の判定案をLayer1が裁定する構造は判断ミスを減らす。
- **不足しやすい点**
  - queue状態機械が弱く、夜間に複数タスクが動くと「今どれがreview待ちか」の視認性が落ちやすい。

**非同期運用評価:** 7.9/10

## 4-2. B案

- **できること**
  - HANDOFF/VERIFY/RESULTの運用があり、翌朝の再開精度が高い。
  - worktree分離とcommand体系により、実行中断後の復帰が実務的。
- **不足しやすい点**
  - queueの状態遷移がCほど「運用OS化」されていないため、多数同時並行時に管理負荷が増える。

**非同期運用評価:** 8.6/10

## 4-3. C案

- **できること**
  - queue state machineが中心のため、朝夕に `blocked/review/stale running` を即点検しやすい。
  - triageセッションの定例化、runsの構造化、reviewゲートにより「続行/中断/やり直し」判断が速い。
- **不足しやすい点**
  - ルール運用が崩れると逆に複雑化するため、初期の運用訓練が必要。

**非同期運用評価:** 9.4/10

## 6. 実務向け提案（最適解は「Cを主軸にAを上位統治として被せる」）

完全に1案を選ぶより、次のハイブリッドが最も強い。

- **骨格:** C案（queue state machine, worktree, runs, review gate）
- **統治理念:** A案（Layer責務・最終意思決定・判定責任）
- **実装テクニック:** B案（settings/hooks/skills/subagents運用）

### 最小導入ステップ（失敗しにくい順）

1. C案の queue ディレクトリと state 遷移だけ先に導入
2. A案の判定責任（Layer2提案→Layer1裁定）を運用ルール化
3. B案の hooks を `PreToolUse` / `Stop` から段階導入
4. 週次で blocked 棚卸しと stale running 整理を固定化

## 7. ユーザー質問への直接回答

### Q1. 進化が激しい中でもベースを崩さず運用できるのはどれか？

**最有力はC案。**
理由は、モデル能力ではなく「queue状態」「worktree境界」「handoff/runs」という**不変の運用構造**に依存しているため。
次点はB案（Claude最適化が強いがベンダー抽象化は弱め）。
A案は理念が強いが、実装規律が不足すると劣化しやすい。

### Q2. 夜間実行→朝/帰宅時のリアルタイム判断・トリアージに最も強いのはどれか？

**C案が最強。**
`blocked/review/running` を状態で即座に切り分けられ、レビュー待ち・再実行・新規投入の判断が短時間で可能。
B案も実務上かなり強いが、C案ほど状態機械中心に設計されていない。
A案は再開思想は良いが、ダッシュボード性が弱い。

---

## 8. 最終提言（意思決定）

- 今すぐ1つ選ぶなら **C案採用**。
- ただし組織運用で責任境界を明確化したいなら、**A案のLayer責務をC案に上書き**する。
- 初期導入負荷が気になるなら、**B案の実装手順を使ってC案へ段階移行**する。

この順で進めると、将来のモデル更新（Claude/Codexの新機能追加）に追随しつつ、夜間バッチ的な非同期開発でも意思決定品質を維持できる。
